\chapter{Contexte et étude du projet}

\section{Introduction}

Ce rapport présente le travail réalisé durant ce stage d'été 2026. Le projet porte sur le développement et la mise en place d'une application complète de gestion de stock incluant une architecture microservices robuste, une infrastructure de monitoring et une pipeline de déploiement continu.

L'objectif principal était de concevoir et déployer une solution complète d'entreprise avec un focus particulier sur la qualité du code, la sécurité et la performance. La solution intègre des technologies modernes comme Docker, Kubernetes (préparation), Jenkins pour l'intégration continue, et des outils de monitoring avec Prometheus et Grafana.

\section{Présentation du projet}

\subsection{Problématique}

Les entreprises de distribution et de logistique font face à plusieurs défis majeurs :

\begin{itemize}
    \item \textbf{Gestion inefficace des stocks} : Manque de visibilité en temps réel sur l'inventaire
    \item \textbf{Erreurs de saisie} : Processus manuels sujets à erreurs
    \item \textbf{Mauvaise traçabilité} : Difficulté de suivre les commandes et les livraisons
    \item \textbf{Absence de monitoring} : Pas de visibilité sur la performance du système
    \item \textbf{Risques de sécurité} : Absence de scanner de vulnérabilités
    \item \textbf{Processus de déploiement non automatisé} : Déploiements manuels et coûteux
\end{itemize}

\subsection{Solution proposée}

Pour résoudre ces problèmes, une application web complète de gestion de stock a été développée avec les composants suivants :

\subsubsection{Architecture Technique}

\begin{itemize}
    \item \textbf{Frontend} : Application React moderne avec TypeScript
    \begin{itemize}
        \item Framework Vite pour le build optimisé
        \item Tailwind CSS pour le styling
        \item Composants UI réutilisables
    \end{itemize}
    
    \item \textbf{Backend} : API RESTful avec Node.js/Express
    \begin{itemize}
        \item ORM Prisma pour la gestion de base de données
        \item PostgreSQL comme base de données principale
        \item Authentification et autorisation
    \end{itemize}
    
    \item \textbf{Containerisation} : Docker pour l'isolation et la portabilité
    \item \textbf{Orchestration} : Préparation pour Kubernetes
    \item \textbf{CI/CD} : Pipeline Jenkins automatisée
    \item \textbf{Monitoring} : Stack avec Prometheus et Grafana
    \item \textbf{Sécurité} : Trivy pour le scan de vulnérabilités
    \item \textbf{Qualité de code} : SonarQube pour l'analyse statique
\end{itemize}

\subsubsection{Fonctionnalités Principales}

L'application propose les fonctionnalités suivantes :

\begin{enumerate}
    \item \textbf{Gestion des produits} : CRUD complet avec catégorisation
    \item \textbf{Gestion des commandes} : Suivi des commandes en temps réel
    \item \textbf{Gestion des clients} : Portefeuille de clients avec historique
    \item \textbf{Gestion des livraisons} : Attribution aux chauffeurs
    \item \textbf{Dashboard analytique} : Visualisations et statistiques en temps réel
    \item \textbf{API REST complète} : Intégrations externes possibles
\end{enumerate}

\section{Conclusion}

La solution proposée répond directement aux enjeux identifiés en fournissant une plateforme moderne, sécurisée et performante. L'architecture choisie garantit la scalabilité et la maintenabilité à long terme.

\section{Architecture DevOps Globale}

Le projet repose sur une infrastructure DevOps complète automatisant le cycle de vie complet de l'application, de la récupération du code à la mise en production avec monitoring continu.

\subsection{Composants Principales}

\begin{itemize}
    \item Jenkins : Orchestration de la pipeline CI/CD
    \item Docker : Conteneurisation des applications
    \item SonarQube : Analyse statique et qualité du code
    \item Trivy : Scan de sécurité des images Docker
    \item Prometheus : Collection des métriques
    \item Grafana : Visualisation et monitoring
    \item PostgreSQL : Base de données persistante
\end{itemize}

\section{Pipeline CI/CD avec Jenkins}

\subsection{Vue d'ensemble}

La pipeline Jenkins est le cœur du système DevOps. Elle automatise :

\begin{enumerate}
    \item Récupération du code depuis GitHub
    \item Analyse de qualité du code
    \item Build des applications
    \item Création des images Docker
    \item Scan de sécurité
    \item Push des images
    \item Déploiement en production
    \item Vérification de santé des services
\end{enumerate}

\subsection{Étapes Détaillées}

\subsubsection{Stage 1 : Clone Repository}

\begin{verbatim}
stage('Clone Repository') {
    steps {
        checkout scm
    }
}
\end{verbatim}

Récupère le dernier code depuis le repository GitHub.

\subsubsection{Stage 2 : SonarQube Analysis}

\begin{verbatim}
stage('SonarQube Analysis - Backend') {
    steps {
        dir('backend') {
            sh '''
                sonar-scanner \
                  -Dsonar.projectKey=stock-backend \
                  -Dsonar.host.url=${SONARQUBE_URL} \
                  -Dsonar.login=${SONARQUBE_TOKEN} \
                  -Dsonar.sources=src
            '''
        }
    }
}
\end{verbatim}

Analyse la qualité du code backend et frontend avec les métriques :
\begin{itemize}
    \item Code Smells
    \item Bugs détectés
    \item Vulnérabilités
    \item Couverture de tests
    \item Duplication de code
\end{itemize}

\subsubsection{Stage 3 : Build Applications}

\begin{verbatim}
stage('Build Backend') {
    steps {
        dir('backend') {
            sh '''
                npm install
                npx prisma generate
            '''
        }
    }
}

stage('Build Frontend') {
    steps {
        dir('frontend') {
            sh 'npm install && npm run build'
        }
    }
}
\end{verbatim}

Compile les applications et génère les artifacts.

\subsubsection{Stage 4 : Docker Build}

\begin{verbatim}
stage('Build Docker Images') {
    steps {
        script {
            sh '''
                docker build -t ${BACKEND_IMAGE} ./backend
                docker build -t ${FRONTEND_IMAGE} ./frontend
            '''
        }
    }
}
\end{verbatim}

Crée les images Docker pour backend et frontend.

\subsubsection{Stage 5 : Security Scan - Trivy}

\begin{verbatim}
stage('Security Scan - Trivy') {
    steps {
        script {
            sh '''
                mkdir -p ${WORKSPACE}/trivy-reports
                
                docker run --rm \
                  -v /var/run/docker.sock:/var/run/docker.sock \
                  -v ${WORKSPACE}/trivy-reports:/reports \
                  aquasec/trivy:0.69.3 image \
                  --skip-db-update \
                  --severity HIGH,CRITICAL \
                  --format json \
                  --output /reports/backend.json \
                  ${BACKEND_IMAGE}
            '''
        }
        archiveArtifacts artifacts: 'trivy-reports/*.json', 
                        allowEmptyArchive: true
    }
}
\end{verbatim}

Détecte les vulnérabilités dans les images Docker :
\begin{itemize}
    \item Vulnérabilités du système d'exploitation
    \item Vulnérabilités des dépendances
    \item Vulnérabilités critiques et hautes
\end{itemize}

Les rapports sont archivés pour traçabilité.

\subsubsection{Stage 6 : Push to Registry}

\begin{verbatim}
stage('Push to Registry') {
    steps {
        script {
            sh '''
                docker push ${BACKEND_IMAGE}
                docker push ${FRONTEND_IMAGE}
            '''
        }
    }
}
\end{verbatim}

Pousse les images validées vers Docker Hub.

\subsubsection{Stage 7 : Deploy to Production}

\begin{verbatim}
stage('Deploy to Production') {
    steps {
        script {
            sh '''
                docker pull ${BACKEND_IMAGE}
                docker pull ${FRONTEND_IMAGE}
                docker compose -f docker-compose.yml up -d
            '''
        }
    }
}
\end{verbatim}

Récupère les images et redémarre les services.

\subsubsection{Stage 8 : Health Check}

\begin{verbatim}
stage('Health Check') {
    steps {
        script {
            sh '''
                curl -f http://localhost:3001/api/health
                curl -f http://localhost:5173
                curl -f http://localhost:9090
                curl -f http://localhost:3000
            '''
        }
    }
}
\end{verbatim}

Vérifie la disponibilité de tous les services.

\section{Containerisation avec Docker}

\subsection{Infrastructure Docker Compose}

Le fichier docker-compose.yml orchestre tous les services :

\begin{verbatim}
version: '3.8'

services:
  backend:
    build: ./backend
    ports:
      - "3001:3001"
    environment:
      DATABASE_URL: postgresql://user:pass@db:5432/db
    depends_on:
      - db
      
  frontend:
    build: ./frontend
    ports:
      - "5173:5173"
    
  db:
    image: postgres:15
    environment:
      POSTGRES_PASSWORD: password
    volumes:
      - postgres_data:/var/lib/postgresql/data
      
  prometheus:
    image: prom/prometheus
    ports:
      - "9090:9090"
    volumes:
      - ./monitoring/prometheus.yml:/etc/prometheus/prometheus.yml
      
  grafana:
    image: grafana/grafana
    ports:
      - "3000:3000"

volumes:
  postgres_data:
\end{verbatim}

\subsection{Dockerfile Backend}

\begin{verbatim}
FROM node:20-alpine

WORKDIR /app
ENV DATABASE_URL=postgresql://dummy@localhost/dummy

COPY package*.json ./
RUN npm install

COPY . .
COPY prisma ./prisma/
RUN npx prisma generate

EXPOSE 3001
CMD ["npm", "run", "dev"]
\end{verbatim}

Optimisations :
\begin{itemize}
    \item Image Alpine (63 MB vs 300+ MB standard)
    \item Multi-stage builds possibles
    \item Variables d'environnement configurables
\end{itemize}

\section{Challenges et Solutions}

\subsection{Challenge 1 : Permissions Docker dans Jenkins}

\textbf{Problème} : Permission denied lors de l'accès à docker.sock

\textbf{Causes} :
\begin{itemize}
    \item User jenkins n'a pas les permissions
    \item Socket non monté correctement
\end{itemize}

\textbf{Solutions implémentées} :
\begin{itemize}
    \item Exécution du conteneur en tant que root
    \item Alternative : Ajout au groupe docker : \verb|usermod -aG docker jenkins|
    \item Montage correct du socket : \verb|-v /var/run/docker.sock:/var/run/docker.sock|
\end{itemize}

\subsection{Challenge 2 : Trivy - Archivage des Rapports}

\textbf{Problème} : Les rapports JSON n'étaient pas générés

\textbf{Causes} :
\begin{itemize}
    \item Chemin workspace incorrect
    \item Répertoire n'existant pas
    \item Docker run avec chemins mal mappés
\end{itemize}

\textbf{Solutions implémentées} :
\begin{itemize}
    \item Utilisation de \verb|${WORKSPACE}| au lieu de \verb|$(pwd)|
    \item Création préalable du répertoire
    \item Mapping correct : \verb|-v ${WORKSPACE}/trivy-reports:/reports|
    \item \verb|allowEmptyArchive: true| pour éviter les erreurs
\end{itemize}

\subsection{Challenge 3 : Prisma Generate en Docker Build}

\textbf{Problème} : DATABASE_URL manquante lors du build

\textbf{Solutions} :
\begin{itemize}
    \item ARG et ENV pour variable dummy
    \item Prisma generate sans connexion réelle
    \item Variable réelle fournie au runtime
\end{itemize}

\subsection{Challenge 4 : LaTeX Compilation}

\textbf{Problème} : Encoding utf-8.def not found

\textbf{Solution} :
\begin{itemize}
    \item Suppression de \verb|\usepackage[utf-8]{inputenc}|
    \item LaTeX moderne utilise UTF-8 par défaut
    \item Nettoyage des contenus dupliqués
\end{itemize}
